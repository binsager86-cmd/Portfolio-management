================================================================================
  1. ARCHITECTURE & TECH STACK
================================================================================

Application:  Cross-platform portfolio management & financial analysis app
Platforms:    Web (Metro bundler / Vercel), iOS (EAS), Android (EAS)

Architecture: Three-tier monorepo
  ┌─────────────────────────────────────────────────────────────┐
  │  mobile-app/  (Expo / React Native)                        │
  │  ├── Expo SDK 54, React 19.1, React Native 0.81.5          │
  │  ├── Expo Router v6 (file-based routing)                   │
  │  ├── Zustand + React Query (state management)              │
  │  └── TypeScript 5.9 (strict mode)                          │
  ├─────────────────────────────────────────────────────────────┤
  │  backend-api/  (FastAPI / Python)                           │
  │  ├── FastAPI 0.111+, Python 3.12                           │
  │  ├── SQLAlchemy 2.0+ ORM + raw SQL helpers                 │
  │  ├── SQLite (dev) / PostgreSQL (prod)                      │
  │  ├── APScheduler for cron jobs                             │
  │  └── Google Gemini AI for PDF extraction                   │
  ├─────────────────────────────────────────────────────────────┤
  │  dev_portfolio.db  (SQLite database — development)         │
  └─────────────────────────────────────────────────────────────┘

Communication: REST API over HTTPS (port 8004 dev, DigitalOcean prod)
Auth:          JWT (access + refresh tokens) with Google OAuth
Version:       1.0.0 / Phase 3

================================================================================
  8. CORE SERVICES & BUSINESS LOGIC
================================================================================

8.1 Portfolio Service (~800 LOC)
────────────────────────────────
The most critical service — all portfolio calculations route through here.

Key Methods:
  get_current_holdings(portfolio)    — Holdings aggregation from transactions
  build_portfolio_table(name)        — Full detailed holdings table (18 columns)
  get_complete_overview()            — Aggregated data across all portfolios
  get_account_balances()             — External account summaries
  calculate_performance(period)      — TWR/MWRR via twr_calculator
  calculate_sharpe_ratio(rf_rate)    — Risk-adjusted returns
  calculate_sortino_ratio(mar)       — Downside-only volatility
  calculate_realized_profit_details()— Per-transaction P&L breakdown
  recalc_portfolio_cash()            — 5-source cash reconciliation
  get_total_portfolio_value()        — Unified value (KWD)

WAC (Weighted Average Cost) Engine — CFA/IFRS-compliant:
  Buy   → cost += purchase_cost + fees; shares += qty
  Sell  → avg = cost/shares; realized_pnl += (sell_value - fees) - avg*qty
  Bonus → shares += qty (zero cost, dilutes avg_cost)

Key Calculations:
  Average Cost:     cost_basis / shares
  Unrealized P&L:   (market_price - avg_cost) × shares
  Realized P&L:     Tracked per sell, cumulative per position
  Portfolio Value:   Stock value (CCY) + USD→KWD conversion via FX
  Cash Balance:      Deposits − Buys + Sells + Dividends − Fees
  Allocation %:      position_value / total_portfolio_value

8.2 Extraction Service (~1000+ LOC)
───────────────────────────────────
AI-powered financial statement extraction from PDFs.

Pipeline:
  1. PDF Preprocessing  — Convert to 250 DPI images via PyMuPDF
  2. Reasoning Pass     — Gemini analyzes with scratchpad arithmetic
  3. Extraction & Audit — AI outputs JSON with internal validation
  4. Verification Step  — Code-level cross-checks (totals vs. sum-of-parts)
  5. Caching            — Content-hash lookup skips reprocessing
  6. Retry              — Up to 3 retries on mismatch

Extracted Data: statement_type (income/balance/cashflow/equity),
  periods, line items, audit checks, confidence metrics

8.3 Integrity Service (~400 LOC)
────────────────────────────────
Financial data verification:
  1. Cash Balance Verification  — Expected vs. Stored
  2. Position Verification      — Aggregate SQL vs. WAC engine
  3. Snapshot Staleness          — Warn if > 3 days old
  4. Anomaly Detection           — Zero prices, negative balances, etc.
  5. Completeness Check          — Orphan symbols, missing deposits

8.4 Price Service (~300 LOC)
───────────────────────────
Stock price updates via yfinance.
  • Kuwait stocks: append .KW suffix
  • Variation mapping (AGILITY → AGLTY)
  • Kuwait quotes in fils → divide by 1000
  • Result tracking: updated/failed/skipped counts

8.5 TWR/MWRR Calculator (~200 LOC)
──────────────────────────────────
  TWR:   Modified Dietz chain-linking (sub-period returns)
  MWRR:  Newton-Raphson root-finding on NPV function
  Both annualized as percentages; requires min 2 snapshots

8.6 FX Service (~150 LOC)
────────────────────────
  In-memory cache with TTL (default 3600s)
  yfinance USD/KWD rate with exponential backoff
  Fallback: DEFAULT_USD_TO_KWD = 0.307190
  Portfolio CCY mapping: KFH/BBYN = KWD, USA = USD

8.7 Cashflow Reconciler (~200 LOC)
─────────────────────────────────
5-source verification:
  Expected = Deposits − Buys + Sells + Dividends − Fees

8.8 Audit Service (~150 LOC)
───────────────────────────
Immutable append-only logging:
  Event types: AUTH_LOGIN, AUTH_LOGIN_FAILED, AUTH_REGISTER,
    AUTH_PASSWORD_CHANGE, AUTH_TOKEN_REFRESH, AUTH_LOCKOUT,
    CASH_*, TRANSACTION_*, STOCK_*, PORTFOLIO_*, ADMIN_*
  Logged data: user_id, action, resource, details (JSON),

  26. ARCHITECTURE PATTERNS & DESIGN DECISIONS
================================================================================

26.1 Backend Patterns
─────────────────────
  • Service Layer Separation: Routes (thin) → Services (logic) → Models (ORM)
  • Dependency Injection: FastAPI Depends() for auth, DB, admin
  • Custom Error Hierarchy: APIError → NotFoundError, AuthError, etc.
  • Soft Deletes: is_deleted + deleted_at (audit trail + recovery)
  • Dual DB Support: SQLAlchemy + raw SQL helpers (SQLite/PostgreSQL)
  • Caching: In-memory with TTL (FX rates 1hr, P/E ratios 24hr)
  • Async Routes: All handlers async def with await dependencies
  • APScheduler: File lock for single instance in multi-worker setup

26.2 Frontend Patterns
──────────────────────
  • File-based Routing: Expo Router with route groups (auth/tabs/onboarding)
  • Server State: React Query (TanStack) — 30s staleTime, 401 no-retry
  • Client State: Zustand stores (auth, theme, preferences)
  • Form State: React Hook Form + Zod validation schemas
  • Error Boundaries: Class-based with reset + retry
  • Responsive Hook: useResponsive() for breakpoint-aware UI
  • Feature Gating: Expertise level controls visible tabs/features
  • Token Management: Proactive refresh + 401 silent refresh interceptor
  • Rate Limiting: Client-side 30 req / 10s throttle

26.3 Cross-Cutting
──────────────────
  • REST API contract between frontend and backend
  • JWT token flow: login → access + refresh → auto-refresh → logout
  • Consistent error envelope: { status, error_code, detail }
  • User-scoped data: All queries filtered by authenticated user_id
  • Admin guard: Backend enforce is_admin, frontend hides admin tab
